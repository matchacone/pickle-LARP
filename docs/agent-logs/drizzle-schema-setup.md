## What Was Built

The full database layer for Pickle All. Drizzle ORM is wired to the Supabase Postgres instance via a pooler connection string. All 5 tables from `schema.md` are defined in `client/lib/db/schema.ts`, a migration was generated and applied to the live database, and both the server-side and browser-side Supabase clients were scaffolded. A SQL trigger runbook was written so that new Supabase Auth signups automatically create a `profiles` row.

## How It Works

1. **DB queries:** any server-side code imports `db` from `client/lib/db/index.ts`. The `db` object is a Drizzle instance connected via the `DATABASE_URL` pooler URL (Transaction mode, port 6543 — required for Vercel serverless).
2. **Auth verification:** RSC and Route Handlers call `await createServerClient()` from `lib/supabase/server.ts`, then call `supabase.auth.getUser()`. This is the only auth path — never use Supabase JS for DB queries.
3. **Client auth/realtime:** client components import `createBrowserClient()` from `lib/supabase/client.ts`.
4. **Profile auto-creation:** the SQL in `supabase_trigger_profiles.sql` must be manually run in the Supabase Dashboard → SQL Editor. It installs a `AFTER INSERT ON auth.users` trigger that inserts a `profiles` row using the email prefix as the default username.

## Key Decisions

- **`timestamptz` is not a named export in drizzle-orm v0.45.** Use `timestamp(name, { withTimezone: true, mode: 'date' })`. A local `tstz()` helper was added to `schema.ts` to avoid repetition.
- **Env key name is `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`**, not the standard `NEXT_PUBLIC_SUPABASE_ANON_KEY`. Both Supabase clients reference this non-standard name. Do not rename it.
- **Profiles trigger is NOT managed by drizzle-kit** — it lives only as a runbook SQL file and must be applied manually. Drizzle-kit cannot manage triggers on `auth.users` (Supabase system schema).
- **`onDelete: 'restrict'` on `invoice.userId` and `invoice.courtId`** — bookings must not silently disappear when a user or court is deleted. Admin must cancel invoices first.
- **`onDelete: 'cascade'` on `reviews`** — reviews are owned content; deleting the user or court removes them automatically.

## Drizzle Schema Changes

All 5 tables created in migration `0000_cool_steel_serpent.sql`:

```typescript
// Shorthand — all timestamps use this
const tstz = (name: string) => timestamp(name, { withTimezone: true, mode: 'date' })

profiles   — id (uuid PK, FK auth.users), username, role CHECK('user'|'admin'), created_at, updated_at
court      — id, court_name UNIQUE, description, created_at, updated_at
item       — id, item_name UNIQUE, created_at
court_item — composite PK (court_id, item_id), FK cascades
invoice    — id, user_id FK restrict, court_id FK restrict, payment_method, payment_total numeric(10,2),
             start_at, end_at, status CHECK('pending'|'confirmed'|'cancelled'|'no_show'), created_at, updated_at
reviews    — id, user_id FK cascade, court_id FK cascade, title nullable, description, created_at, updated_at
```

Command run: `npx drizzle-kit generate` → `lib/db/migrations/0000_cool_steel_serpent.sql`
Then: `npx drizzle-kit migrate` → applied to live Supabase DB.

## Known Gaps and Gotchas

- **No RLS policies created yet.** The tables exist but Supabase Row-Level Security policies have not been defined. Until RLS is configured, all data is readable/writable by any authenticated connection. This must be addressed before production.
- **Profiles trigger not verified in prod** — must be manually run in Supabase Dashboard. If omitted, Google OAuth sign-up will create `auth.users` rows with no matching `profiles` row, causing 500 errors on any query that joins `profiles`.
- **`drizzle.config.ts` is in `client/` root** — always run `npx drizzle-kit generate/migrate` from inside the `client/` directory, not repo root.
- **No `unique(user_id, court_id)` constraint on `reviews`** — the DATA_MODELS.md recommends it but it was not applied in the migration. A future agent must add this via a new migration.

## How to Extend This

- To add a new table: edit `client/lib/db/schema.ts`, run `npx drizzle-kit generate` from `client/`, then `npx drizzle-kit migrate`.
- To add a column: same process. Also update `docs/DATA_MODELS.md` and `docs/schema.md`.
- Use the `schema-sync` skill for any schema change to propagate it through all downstream docs.
- Do not edit files in `client/lib/db/migrations/` manually.
